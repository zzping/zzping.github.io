# Mybatis 的 SQL 映射文件

MyBatis 的真正强大在于它的语句映射，SQL 就是写在SQL 映射文件中的。

# 标签概览

SQL 映射文件中可以写如下的标签（按照应被定义的顺序列出）

  • cache – 该命名空间的缓存配置。

  • cache-ref – 引用其它命名空间的缓存配置。

  • resultMap – 描述如何从数据库结果集中加载对象，是最复杂也是最强大的元素。

  • parameterMap – 已被废弃

  • sql – 可被其它语句引用的可重用语句块。

  • insert – 映射插入语句。

  • update – 映射更新语句。

  • delete – 映射删除语句。

  • select – 映射查询语句。

# （2）获取新增记录的主键

对于支持自增主键的数据库（MySQL、SQL Server）

<!--
useGeneratedKeys: （仅适用于 insert 和 update）这会令 MyBatis 使用 JDBC 的 getGeneratedKeys 方法来取出由数据库内部生成的主键（比如：像 MySQL 和 SQL Server 这样的关系型数据库管理系统的自动递增字段），默认值：false。

keyProperty: （仅适用于 insert 和 update）指定能够唯一识别对象的属性，MyBatis 会使用 getGeneratedKeys 的返回值或 insert 语句的 selectKey 子元素设置它的值，默认值：未设置（unset）。如果生成列不止一个，可以用逗号分隔多个属性名称。
-->


可以通过设置 insert 标签的 useGeneratedKeys 属性等于 true 来获取自增主键, 然后通过 keyProperty 将自增主键绑定到对应 JavaBean 的属性上。(底层原理是使用 JDBC 的 getGeneratedKeys 方法来取出由数据库内部生成的主键)。单个新增和批量新增都是通过这种方式获取自增主键。


<!-- 新增单个, 获取自增主键 -->
<insert id="save" useGeneratedKeys="true" keyProperty="id">
insert into t_user(last_name, gender, email)
values(#{lastName}, #{gender}, #{email})
</insert>


<!-- 批量新增, 获取自增主键 -->
<insert id="saveBatch" useGeneratedKeys="true" keyProperty="id">
insert into t_user(last_name, gender, email)
values
<foreach collection="list" item="item" separator=",">
  (#{item.lastName}, #{item.gender}, #{item.email})
</foreach>
</insert>

对于不支持自增主键的数据库（Oracle）

useGeneratedKeys、keyProperty 两个属性只适用于支持自增主键的数据库。对于不支持自增主键的数据，例如 Oracle 则需要采用其他方法(selectKey)。

<!-- 新增单个 -->

<selectKey keyProperty="id" order="BEFORE" resultType="int">
  select user_seq.nextval from dual 
</selectKey>

values(#{id}, #{lastName}, #{gender}, #{email})

# （2）多个参数

多个参数： mybatis 会做特殊处理。多个参数会被封装成 一个map。

  • map 的 key： param1...paramN
  • map 的 value：传入的参数值

所以这个时候取值的方式可以使用 #{param1}、#{param2} 这种方式来取值。但是这种名字不够直观，所以可以通过给参数添加注解 @Param 来指定参数名，指定好参数名后就可以通过 @Param 指定的参数名来取值。

（1）多个参数，并且都不带 @Param 注解

<select id="getUserByIdAndLastName" resultType="com.zzp.entity.User">
	select * from t_user where id = #{id} and last_name = #{lastName}
</select>

上述方式取值直接报错： BindingException: Parameter 'id' not found. Available parameters are [arg1, arg0, param1, param2]

两种取值方式：
select * from t_user where id = #{arg0} and last_name = #{arg1}
select * from t_user where id = #{param1} and last_name = #{param2}

# （3）# 和 $ 两者的区别【重要】

首先，#{} 和 ${} 都是用于取值， 他们有不同之处：

${}： 取出的值直接拼装在 sql 语句中，会有 SQL 注入的问题

#{}： 占位符的方式，会通过 PreparedStatement 进行对 SQL 预编译，可以防止 SQL 注入

select * from tbl_employee where id=${id} and last_name=#{lastName}

select * from tbl_employee where id=10 and last_name=?
1
2
3

两者使用场景：

（1）#{}

大多情况下，我们去参数的值都应该去使用#{}, 比如各种条件参数

select * from t_user where id = #{id}
1
2
3

（2）${}

原生jdbc不支持占位符的地方我们就可以使用 ${} 进行取值, 比如表名、字段名、排序等场景。

select * from ${tableName} where id = #{id}
select * from t_user order by ${orderBy} ${order}

# resultMap（自定义结果集映射）

# （1）resultMap 简介

resultType 是自动封装相关的。查出数据后，通过 resultType 指定要封装成什么类型的数据。mybatis 会自动封装成对应数据类型。如果查出的列名和Java的属性名不一样，一开始是封装不成功的，有多种解决方案：

  • 查询 SQL 中写别名
  • 如果列名和Java属性名符合驼峰命名规则，则可以在配置文件中开启驼峰命名自动映射。
  • 使用 resultMap 自定义结果集映射(自己来定义列名和Java属性名的一一映射关系)

自动映射相关的设置

autoMappingBehavior:
指定 MyBatis 应如何自动映射列到字段或属性。 NONE 表示关闭自动映射；PARTIAL 只会自动映射没有定义嵌套结果映射的字段。 FULL 会自动映射任何复杂的结果集（无论是否嵌套）。默认值：PARTIAL


mapUnderscoreToCamelCase：
是否开启驼峰命名自动映射，即从经典数据库列名 A_COLUMN 映射到经典 Java 属性名 aColumn。默认值 false

# （6）resultMap 鉴别器 discriminator

resultMap 中的 discriminator 可以判断某一列的值，根据这一列的值的不同使用不同的封装规则。

实际开发中，这里的根据不同的值进行不同的逻辑处理在 Java 代码中实现比较好，持久层一般不做业务逻辑相关的功能。

# （7）resultType 和 resultMap 对比总结

（1）resultType 和 resultMap 功能都差不多，都是映射列名和属性名，后者更强大，可以自定义。

（2）resultMap 要求列名和属性名要相同，resultMap 则没有限制。
