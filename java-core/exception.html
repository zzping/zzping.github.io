# Java 异常体系

# 异常简介

异常是程序在"编译"或者"运行"的过程中可能出现的问题，异常不处理会导致程序出现中断运行。

异常是指由于各种不期而至的情况，导致程序中断运行的一种指令流,如：文件找不到、非法参数、网络超时等。为了保证正序正常运行，在设计程序时必须考虑到各种异常情况，并正确的对异常进行处理。

Java 中的异常是通过面向对象的方式进行管理的, 对于每一种类型的异常都会有一个异常类。

# 异常分类

Java 中定义了 java.lang.Throwable 作为所有异常类的基类。并且将异常分为了两大类：

  • Error（错误）： 表示系统内部错误，通常是 Java 虚拟机内部出现错误而崩溃。例如： OutOfMemoryError（内存溢出错误）、VirtualMachineError（虚拟机错误）。这种情况仅凭程序自身是无法处理的，在程序中也不会对 Error 错误进行捕捉和抛出。
  • Exception（异常）： 表示程序本身可以处理的异常，具体按照异常出现的时机又可以分为两类
    • 编译时异常：程序编译期间编译器就能发现的异常，要求代码中必须处理，否则程序无法编译通过。除了运行时异常（RuntimeException 本身及其子类），其他异常都是编译时异常。常见的编译时异常比如 IOException、SQLException 等。
    • 运行时异常：程序运行过程中可能出现的异常，RuntimeException 及其所有子类都属于运行时异常。运行时异常描述的是一种可能性，在程序运行过程中可能出现异常，也可能不会出现。常见的运行时异常比如 NullPointException、ClassCastException、ArrayIndexOutOfBoundsException 等。

# 异常处理机制

# （1）默认的异常处理机制

程序中如果出现了异常，我们没有进行任何处理，默认的处理机制是： 会在出现异常的代码处创建对应的异常对象（比如 ArithmeticException），然后从这里抛给往上方法的调用者，最终抛给 JVM 虚拟机。JVM 收到异常对象后，现在控制台输出异常堆栈信息，然后 JVM 虚拟机结束程序（程序退出）。当然后面的代码也就不会执行了。

# （2）编译时异常的处理

编译时异常必须处理，否则编译无法通过，处理方式主要分为两种：

方式一： 抛出异常（throws）

在出现异常的地方通过 throws 层次往上抛出，谁都不处理。这种方式不太好，因为如果异常一旦出现了，会一直往上抛给调用者，直到抛给虚拟机，最终虚拟机会终止程序。

throws 写在方法声明后面, throws 后面异常类名，可以写多个异常类名

@Test
public void test01() throws FileNotFoundException, ParseException {
   FileInputStream fileInputStream = new FileInputStream("d:/a.txt");
   SimpleDateFormat sdf = new SimpleDateFormat("yyyy-MM-dd HH:mm:ss");
   Date date = sdf.parse("2021-05-25 00:00:00");
}

# （3）运行时异常的处理

运行时异常在编译阶段不会报错，在运行阶段才会出现。

运行时异常一般是程序中有一定逻辑问题（比如 NullPointerException、ArithmeticException: / by zero），是不应该出现的，应该通过修改程序逻辑从而解决异常。

比如下面这一段代码在运行时肯定会出现 ArrayIndexOutOfBoundsException 的异常，这种运行时的异常出现的原因一般是我们的程序逻辑不够严谨而导致的，应该通过检查修改程序的正确性来完成运行时异常的处理。

int[] array = {1,2,3,4,5};
System.out.println(array[5]); // java.lang.ArrayIndexOutOfBoundsException: 5

# （4）异常处理的5个关键字

Java 异常处理涉及到有 5 个关键字： try、catch、finally、throw、throws

try、catch、finally： 这三个关键字通常一起用于捕获异常进行处理

  • try 里面跟的是可能出现异常的语句或者代码块，后面至少要跟 catch 或者 finally 中的一种。
  • catch 中指定异常的种类，可以有多个异常，出现异常后会匹配 catch 中的异常类型。
  • finally 中的代码一定会执行（不管是否出现异常），所以通常在 finally 中释放资源、关闭连接等。
public void test01() {
    try {
        FileInputStream fileInputStream = new FileInputStream("d:/a.txt");
        SimpleDateFormat sdf = new SimpleDateFormat("yyyy-MM-dd HH:mm:ss");
        Date date = sdf.parse("2021-05-25 00:00:00");
    } catch (FileNotFoundException e) {
        e.printStackTrace();
    } catch (ParseException e) {
        e.printStackTrace();
    } finally {
        // ...
    }
}

throw、throws 都是用于抛出异常，但是他们之间有区别：

// throws + 异常类名

# （5）面试题及小结

面试题1： final、finally、finalize 三者的区别？

  • final： java 中的一个关键字，可以修饰类、成员变量、成员方法
    • 修饰类，表示该类不能被继承
    • 修饰成员变量，表示该变量是一个常量
    • 修饰成员方法，表示该方法不能被重写
  • finally： 异常处理的一个关键字，通常和 try-catch 一起使用，在 finally 中主要用于释放资源。
  • finalize： 是 Object 类中的一个方法，用于垃圾回收。

面试题2：如果 catch 里面有 return 语句，finally 里面的代码还会执行吗？如果会，请问是在 return 前还是在 return 后？

  • 答案： 会，在 return 之前
  • 解析如下：
public class ConstantPoolTest {

    public static void main(String[] args) {
        System.out.println(method1()); // 30
        System.out.println(method2()); // 40
    }

    public static int method1() {
        int a = 10;
        try {
            System.out.println(a/0);
            a=20;
        } catch (ArithmeticException ex) {
            a = 30;
            return a;
            // 程序到 return a 这一步的时候，实际上已经不是 return a 而是 return 30,
            // 这个返回路径已经形成了,但是发现还有 finally，于是继续去执行 finally 中
            // 的代码 a = 40，执行完后又回到之前的返回路径 return 30, 所以最终结果返回的是 30
        } finally {
            a = 40;
        }
        return a;
    }
    
    public static int method2() {
        int a = 10;
        try {
            System.out.println(a / 0);
            a = 20;
        } catch (ArithmeticException ex) {
            a = 30;
            return a;
        } finally {
            a = 40;
            return a; // 不要在 finally 中 return, 否则方法返回值永远是这里 return 的值
        }
    }
}

# 自定义异常

Java 中已经为常见的异常都定义的对应的异常类，但实际开发中我们很多时候想要针对某种业务问题定义自己的异常，这时候就需要自定义异常。

# （1）自定义编译时异常

步骤：

A： 定义一个异常类，继承 Exception， 重写构造方法

public class MyException extends Exception {

   public MyException() {
   }

   public MyException(String message) {
       super(message);
   }
}

# （3）小结

一般开发中，自定义运行时异常比较多（自定义异常继承 Runtime Exception），因为如果自定义编译时异常，需要通过 throws 继续向上抛出或者自行处理，调用多的情况下比较麻烦。

# 阿里开发手册 - 异常处理

# 1. 异常不要用来做流程控制，条件控制。

异常设计的初衷是解决程序运行中的各种意外情况，不要通过异常来进行流程控制或者条件控制。因为异常的处理效率比条件判断方式要低很多，并且这样是将程序的各种意外和正常的流程跳转混淆，意义不明。

# 2. 不要在 finally 块中使用 return

finally 块中的 return 返回后方法结束执行，不会再执行 try 块中的 return 语句。因此方法的返回值总是 finally 中的返回值，try 当中的返回值没有效果。
